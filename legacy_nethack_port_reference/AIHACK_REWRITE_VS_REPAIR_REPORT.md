# AIHack 유지보수 vs Rust 맞춤 재구현 분석 리포트

작성일: 2026-04-28

## 질문

현재 AIHack은 NetHack 3.6.7 C 소스를 Rust로 옮긴 뒤 AI 연동을 목표로 하는 프로젝트다. 핵심 고민은 다음 두 가지다.

1. C 원본 구조를 많이 가져온 결과, Rust/ECS 런타임 borrow 충돌이나 데드락성 문제가 여러 곳에서 발생한다.
2. NetHack의 알고리즘과 동작 방식이 세밀하게 구현되지 못한 부분이 많다.

따라서 이 코드베이스를 계속 보수하며 하나씩 고치는 것이 나은지, 아니면 현재 파일/문서/구조만 참고 자료로 남기고 원본 NetHack과 다시 비교해 Rust에 맞는 게임 엔진으로 재구현하는 것이 나은지 판단한다.

## 결론

권장 방향은 **전면 폐기가 아니라, 현재 코드를 "참고 구현 및 테스트 자산"으로 보존한 뒤 핵심 런타임을 Rust 맞춤 아키텍처로 재구성하는 하이브리드 재구현**이다.

현재 코드베이스를 그대로 보수하는 방식은 단기적으로는 빠르지만, 장기적으로는 같은 종류의 문제가 반복될 가능성이 높다. 이유는 버그가 개별 함수의 실수라기보다 다음 구조에서 나온다.

- 실제 게임 실행 경로가 `game_loop.rs`의 대형 직접 오케스트레이션에 집중되어 있다.
- 별도의 `turn_engine.rs`가 존재하지만 실제 몬스터/레벨 이벤트는 TODO 상태이고, 현재 실행 루프와 통합되지 않은 별도 경로다.
- `self.game.grid`와 `resources.Grid` 같은 이중 상태가 이미 문제로 기록되어 있으며, 코드에서도 매 턴 복사/역동기화로 보정하고 있다.
- 문서상 100% 이식 주장과 실제 미구현/부분 구현 목록이 공존한다.
- 많은 시스템이 단위 테스트는 통과하지만, 안정화 문서가 스스로 "ECS와 연결되지 않은 섬 코드"를 핵심 리스크로 지적한다.

따라서 목표가 "NetHack의 완전한 Rust 포트"라면 유지보수 지속도 가능하지만 비용이 매우 높다. 목표가 "AI가 연결 가능하고 안정적으로 실행되는 NetHack 계열 게임"이라면, **원본 NetHack 호환성을 일부 낮추더라도 Rust-native 런타임을 새로 세우는 쪽이 더 낫다.**

## 현재 상태 요약

직접 확인한 현재 저장소 상태:

- Rust 소스 파일: 438개
- Rust 소스 라인: 약 195,354줄
- 테스트 표식 검색: 4,578개
- `cargo test`: 4,177 passed, 0 failed
- `cargo test --lib`: 라이브러리 타깃이 없어 실패
- 현재 작업 트리는 이미 여러 파일이 수정된 상태였으므로, 이 분석은 기존 변경을 되돌리거나 수정하지 않았다.

테스트 통과는 중요한 자산이다. 하지만 이 테스트들은 현재 구조상 "순수 함수/부분 시스템의 정확성"을 강하게 보장하는 반면, "실제 플레이 루프에서 모든 기능이 연결되어 안정적으로 동작한다"는 보장은 약하다.

## 근거

### 1. 문서가 스스로 핵심 위험을 인정한다

`STABILIZATION_ROADMAP.md`는 LLM 연동을 보류해야 한다고 명시한다. 이유는 "로직 데드락 vs LLM 오류 구분 불가"이며, 결정론적 동작을 먼저 안정화해야 한다는 판단이다.

또한 같은 문서는 177,229줄 중 상당수가 Pure Result 패턴으로 이식되어 ECS와 연결되지 않은 "섬 코드"일 수 있고, 실제 `execute_turn_systems()`에 등록된 시스템은 약 25개뿐이라고 적는다. 테스트에서는 단일 시스템만 실행되지만, 실제 schedule 실행에서는 Legion 런타임 borrow checker가 패닉을 낼 수 있다는 점도 기록되어 있다.

관련 위치:

- `STABILIZATION_ROADMAP.md:17`
- `STABILIZATION_ROADMAP.md:34-45`
- `STABILIZATION_ROADMAP.md:137-141`

이는 사용자가 말한 "Rust 특성상 아무 데서나 데드락이 튄다"는 체감이 단순한 오해가 아니라, 프로젝트 내부 문서에도 남아 있는 구조적 리스크임을 보여준다.

### 2. 실제 실행 루프가 단일 엔진으로 정리되어 있지 않다

`game_loop.rs`는 입력 처리, 상태 전환, 액션 큐 배출, ECS 스케줄 실행, Grid 역동기화, EventQueue 후처리, 레벨 변경, 사망 체크를 한 파일 안에서 직접 조율한다.

특히 매 턴 `self.game.grid`를 `resources`에 넣고, 시스템 실행 후 다시 `resources.Grid`를 `self.game.grid`로 복사한다. 주석은 이 역동기화가 없으면 문 열기, 함정 발동 등이 화면에 반영되지 않는다고 설명한다.

관련 위치:

- `src/game_loop.rs:79-111`
- `src/game_loop.rs:1010-1065`
- `src/game_loop.rs:1318-1360`

이 구조는 "동작하게 만들기 위한 보정"으로는 타당하지만, AI 연동을 위한 안정적인 상태 스냅샷/리플레이/시뮬레이션 기반으로는 취약하다. AI가 들어오면 매 턴 상태를 읽고 설명하거나 정책을 제안해야 하는데, 상태의 원천이 둘 이상이면 오류 원인 추적이 어려워진다.

### 3. 새로운 턴 엔진은 있지만 실제 핵심 경로와 분리되어 있다

`turn_engine.rs`에는 `TurnPhase`, `TurnContext`, `run_turn_systems()`가 정의되어 있다. 그러나 `MonsterActions`와 `LevelEvents`는 TODO이고, 실제 `game_loop.rs`의 대형 Legion schedule과 직접 연결된 구조가 아니다.

관련 위치:

- `src/core/systems/turn_engine.rs:44-60`
- `src/core/systems/turn_engine.rs:62-98`
- `src/core/systems/turn_engine.rs:100-110`

이는 재구현의 씨앗이 이미 코드 안에 있다는 의미이기도 하다. 다만 현재 상태에서는 기존 루프를 대체할 만큼 성숙한 엔진이 아니다.

### 4. 문서상 "100% 이식"과 실제 부분 구현 목록이 충돌한다

`IMPLEMENTATION_SUMMARY.md` 상단에는 v2.23.0, 85.8% 이식률이 남아 있고, 아래에는 v2.41.0 100% 이식 완료가 기록되어 있다. 그러나 같은 문서의 모듈 표는 `zap.rs` 39.5%, `trap.rs` 35.4%, `inventory.rs` 31.4%, `apply.rs` 33.9% 같은 부분 이식률을 보여준다. 그리고 "현재 미구현 핵심 시스템"으로 `cmd.c`, `invent.c`, `sp_lev.c`, `trap.c`, `zap.c`, `shk.c` 나머지 이식을 우선순위 높음으로 적고 있다.

관련 위치:

- `IMPLEMENTATION_SUMMARY.md:3-11`
- `IMPLEMENTATION_SUMMARY.md:21-35`
- `IMPLEMENTATION_SUMMARY.md:173-184`
- `IMPLEMENTATION_SUMMARY.md:186-203`

따라서 "100% 이식"은 실제 NetHack 동작 완성도보다는 특정 단계의 산출 완료 선언에 가깝다. NetHack 정합성을 계속 추구하려면 상당한 검증 부채가 남아 있다.

### 5. AI 코드도 아직 게임 전체 상태를 정확히 반영하지 못한다

`src/core/systems/ai/core.rs`에는 트랩 위치 수집 TODO, 몬스터 간 공격 미구현, AC 10 placeholder 등이 남아 있다.

관련 위치:

- `src/core/systems/ai/core.rs:109-130`
- `src/core/systems/ai/core.rs:210-223`

AI 연결을 목표로 할 때 가장 중요한 것은 "AI가 보는 상태가 게임 규칙과 일치한다"는 점이다. 현재 AI/몬스터 로직 일부는 실제 상태를 충분히 끌어오지 못하고 있으므로, 이 위에 LLM이나 외부 정책 모델을 붙이면 원인 분리가 더 어려워질 가능성이 높다.

### 6. ActionQueue는 개선되었지만 런타임 계약이 충분히 좁지 않다

`ActionQueue`는 `GameAction` enum과 타입별 pop 메서드를 제공한다. 다만 실행 루프에서는 큐를 모두 drain한 뒤 각 타입별 `Option<Action>` 리소스에 넣는 방식이 남아 있다. 이 방식은 단일 턴에 여러 액션이 겹칠 때 처리 순서와 덮어쓰기 규칙을 별도로 엄격히 검증해야 한다.

관련 위치:

- `src/core/action_queue.rs:1-39`
- `src/core/action_queue.rs:41-111`
- `src/game_loop.rs:1435-1465`

큐 자체는 좋은 방향이지만, AI 연결용 런타임이라면 "턴 입력 -> 명령 검증 -> 상태 변경 -> 이벤트 산출"이 하나의 명시적 트랜잭션처럼 보이는 구조가 더 적합하다.

## 선택지 비교

### 선택지 A: 기존 코드 보수 지속

장점:

- 이미 195k 라인 규모의 기능 코드와 4,177개 통과 테스트가 있다.
- NetHack 원본의 수많은 특수 규칙을 부분적으로라도 보존하고 있다.
- 던전, 몬스터, 아이템, UI, 저장, 이벤트 등 많은 도메인 타입이 이미 정의되어 있다.
- 단기적으로는 특정 패닉이나 AccessDenied를 고치는 속도가 빠르다.

단점:

- `game_loop.rs` 중심의 직접 조율 구조가 계속 커질 가능성이 높다.
- 상태 원천이 `self.game.*`, Legion `World`, `Resources`, `ActionQueue`, `EventQueue`로 분산되어 있다.
- 문서와 실제 구현 상태의 불일치 때문에 "무엇이 진짜 완성인지" 판단 비용이 크다.
- NetHack 정합성을 끝까지 맞추려면 원본 비교, E2E, snapshot, FFI shadow testing 등 별도 검증 체계가 필요하다.
- AI 연동 후 문제가 생겼을 때 게임 로직 문제인지 AI 문제인지 분리하기 어렵다.

적합한 경우:

- 목표가 "NetHack 3.6.7 Rust 포트" 자체일 때
- 원본 동작과의 호환성이 최우선일 때
- 장기간에 걸쳐 기존 구조를 안정화할 충분한 시간이 있을 때

### 선택지 B: 전면 재구현

장점:

- Rust에 맞는 단일 상태 모델과 명시적 턴 트랜잭션을 처음부터 설계할 수 있다.
- AI 관찰/행동 인터페이스를 엔진의 1급 API로 만들 수 있다.
- Legion 런타임 borrow 충돌, 이중 Grid, 분산 큐 같은 현재의 구조적 리스크를 크게 줄일 수 있다.
- NetHack 완전 복제 대신 "NetHack 계열의 안정적 AI 실험 게임"으로 범위를 줄일 수 있다.

단점:

- 기존 테스트와 구현을 그대로 버리면 수년치 도메인 지식 손실이 크다.
- NetHack의 세부 규칙을 다시 선별해야 한다.
- 당분간 기능 수는 줄어든다.
- 새 엔진도 검증 체계를 잘못 잡으면 현재와 같은 문제가 다시 생길 수 있다.

적합한 경우:

- 목표가 "AI가 안정적으로 연결 가능한 게임"일 때
- 원본 NetHack과 약간 달라져도 괜찮을 때
- 장기 유지보수성과 디버깅 가능성이 완전 호환성보다 중요할 때

### 선택지 C: 하이브리드 재구현

권장안이다.

핵심은 기존 코드를 폐기하지 않고 다음처럼 역할을 바꾸는 것이다.

- 기존 `src/core/**/_ext.rs`: 규칙 참고 자료 및 순수 함수 후보
- 기존 테스트: 회귀 테스트/규칙 문서 역할로 보존
- 기존 문서: 구현 범위와 과거 의사결정의 참고 자료
- 새 런타임: Rust-native 단일 턴 엔진으로 별도 구성
- NetHack 원본: 동작 정합성의 기준 자료

즉, "현재 게임을 계속 땜질"하는 대신 "현재 코드를 레퍼런스 라이브러리처럼 다루고, 실행 엔진만 새로 세우는 방식"이다.

## 권장 아키텍처 방향

새 런타임은 다음 원칙을 가져야 한다.

1. 단일 상태 원천

   `GameWorld` 또는 `GameSession` 하나가 현재 층, 엔티티, RNG, 턴, 로그, 이벤트를 소유한다. 렌더러와 AI는 읽기 전용 view/snapshot을 받는다. `self.game.grid`와 `resources.Grid`처럼 복제본을 양방향 동기화하지 않는다.

2. 명시적 턴 트랜잭션

   입력은 `CommandIntent`가 되고, 엔진은 이를 `ValidatedAction`으로 바꾼 뒤 `TurnOutcome`을 반환한다.

   예:

   ```text
   CommandIntent -> validate -> apply -> events -> render/ai snapshot
   ```

3. 엔진과 UI 분리

   `egui`/`ratatui`는 `GameSnapshot`만 읽고 `CommandIntent`만 보낸다. UI 파일이 직접 World/Grid/Resources를 수정하지 않는다.

4. AI API 선설계

   AI가 받는 정보는 `Observation`으로 고정한다. AI가 낼 수 있는 행동은 `ActionSpace`로 제한한다. LLM 텍스트 생성은 게임 로직 뒤의 narrative layer에만 붙인다.

5. NetHack 호환성은 계층화

   모든 NetHack 규칙을 한 번에 옮기지 않는다. `core rules`, `advanced rules`, `NetHack quirks`로 나누고, 각 규칙은 테스트 가능한 작은 모듈로 둔다.

6. 원본 비교는 "선별적 golden test"로 제한

   모든 C 동작을 1:1로 맞추려 하지 말고, 이동/전투/아이템/던전 생성/몬스터 AI의 핵심 시나리오만 golden snapshot으로 잡는다.

## 실행 로드맵

### Phase 0: 보존 및 경계 확정

- 기존 코드를 `legacy_reference`로 보는 방침을 문서화한다.
- 현재 테스트 중 엔진 독립적인 순수 함수 테스트와 실제 런타임 결합 테스트를 분류한다.
- "NetHack 완전 포트"가 목표인지 "AIHack 독립 게임"이 목표인지 README와 로드맵의 표현을 정리한다.

### Phase 1: 최소 Rust-native 엔진 작성

범위는 작게 잡는다.

- 단일 층
- 플레이어 이동
- 벽/문/계단
- 몬스터 1~2종
- 기본 근접 전투
- 아이템 줍기/사용 1~2종
- deterministic RNG
- `Observation`/`ActionSpace`/`TurnOutcome`

이 단계의 성공 기준은 "1000턴 자동 시뮬레이션 무패닉"보다 더 좁고 강해야 한다.

- 같은 seed에서 완전히 같은 event log
- 모든 상태 변경이 `TurnOutcome`에 나타남
- AI 없이 headless 실행 가능
- UI 없이 테스트 가능

### Phase 2: 기존 자산 흡수

기존 코드에서 가져올 만한 것은 순수 규칙 함수와 데이터 정의다.

- 아이템/몬스터 TOML 데이터는 유지 가능
- `ItemKind`, `MonsterKind` 같은 enum은 유지 후보
- 전투 공식, RNG 유틸, 이름 생성 등 순수 함수는 선별 흡수
- Legion schedule에 강하게 묶인 시스템은 직접 이식하지 말고 규칙 단위로 재작성

### Phase 3: NetHack 핵심 시나리오 재구성

아래 순서가 적합하다.

1. 이동/시야/문/계단
2. 근접 전투와 사망
3. 인벤토리 문자 슬롯
4. 원거리 투척/마법봉
5. 함정
6. 던전 생성
7. 상점/기도/특수 레벨

각 단계마다 기존 테스트를 재사용하거나 snapshot 테스트로 바꾼다.

### Phase 4: AI 연동

LLM은 마지막에 붙인다.

- 1차: AI 관찰 JSON과 deterministic bot
- 2차: scripted policy
- 3차: LLM narrative only
- 4차: 제한된 LLM decision support

LLM이 직접 게임 상태를 수정하지 않게 해야 한다.

## 최종 판단

현재 코드베이스는 버릴 코드가 아니다. 하지만 현재 실행 구조를 그대로 유지한 채 하나씩 고치는 방식은 AIHack의 최종 목표와 잘 맞지 않는다.

가장 실용적인 판단은 다음이다.

> **기존 프로젝트를 "NetHack 규칙과 테스트의 광산"으로 유지하고, 실제 플레이 가능한 AIHack 런타임은 Rust-native 엔진으로 새로 세운다.**

이 접근은 기존 작업을 존중하면서도, Rust에서 가장 문제가 되는 런타임 borrow 충돌, 상태 이중화, UI/엔진 결합, 미연결 섬 코드 문제를 근본적으로 줄인다.

## 남은 불확실성

- 실제 `cargo run` GUI 경로에서 현재 어느 커맨드까지 안정적으로 동작하는지는 이번 분석에서 직접 플레이 검증하지 않았다.
- 원본 NetHack C 소스와 함수별 diff/동작 비교는 수행하지 않았다.
- 현재 작업 트리에 사용자 변경이 많아, 이 분석은 HEAD 기준과 작업트리 기준이 섞여 있을 수 있다.
- 테스트는 통과하지만 경고가 매우 많다. 특히 unused 변수는 미연결/미구현 로직의 신호일 수 있으므로 별도 감사가 필요하다.
