## 1. C→Rust 이식 교훈

### 1.1 C 구조 직역은 반드시 실패한다
- **교훈**: 원본 C의 전역 변수, goto, 매크로, 포인터 연결 리스트를 그대로 Rust로 번역하면 `unsafe`와 `static mut`가 난무하여 Rust의 장점이 사라진다.
- **해결**: "100% 이식"의 의미를 **동작(Behavior)의 충실한 재현**으로 정의하고, **코드 구조(Structure)의 1:1 번역**을 명시적으로 금지했다.
- **결과**: 프로젝트 전체에서 `unsafe` 블록 0개, `static mut` 0개, `todo!()` 0개 달성.

### 1.2 초기 20%에서 아키텍처를 확립해야 한다
- **교훈**: AI 에이전트는 기존 패턴을 따라가는 특성이 강하다. 이식률 20% 시점에서 올바른 구조(ECS, enum, Pure Result)를 잡아두면, 이후 80%의 이식도 자동으로 그 구조를 따른다.
- **실증**: 37% 시점에서 `unsafe` 0개, `static mut` 0개 — 초기 구조화가 성공적이었음을 확인.

### 1.3 Pure Result 패턴 — C→Rust 포팅의 새로운 방법론
- **교훈**: C2Rust 트랜스파일러 대신, 알고리즘을 순수 함수로 추출하여 부작용 없이 결과만 반환하는 방식이 가장 안전하다.
- **패턴**: (1) 알고리즘 추출 (2) 구조체 입력 캡슐화 (3) Pure 함수 반환 (4) ECS 분리 적용
- **정량적 성과**: unsafe 0건, unwrap 0건, 모든 모듈 1회 수정 내 통과.

## 2. ECS 아키텍처 교훈

### 2.1 Gather-Apply 패턴의 필수성
- **교훈**: Legion ECS에서 동일 SubWorld에 대한 동시 가변 대여가 불가능한 것은 Rust의 고질적 제약이다.
- **해결**: 데이터를 읽는 과정(Gather)과 수정하는 과정(Apply)을 엄격히 분리하는 패턴을 전 시스템에 적용.

### 2.2 God Object 방지 — TurnContext 도메인 분리
- **교훈**: C의 전역 변수를 대체하기 위해 도입한 `TurnContext`가 비대해지면 안티패턴이 된다.
- **해결**: `CombatContext`, `MovementContext` 등 도메인별 분리로 결합도를 낮췄다.

### 2.3 DeferredCommand — 동시 수정 충돌 해소
- **해결**: `DeferredCommand` 객체를 반환 후 Apply 단계에서 일괄 처리. FIFO 순서 우선.

### 2.4 DeathResults → CommandBuffer 전환
- **해결**: Legion의 `CommandBuffer` 패턴을 도입하여 Entity 생성 명령을 큐에 담아 자동 반영.

## 3. 리팩토링 교훈

### 3.1 main.rs 174KB → 5KB (78% 축소)
- **해결**: Rust의 `impl` 블록 분산 기능을 활용. 로컬 변수를 구조체 필드로 승격하여 분리된 메서드 간 데이터 공유.

### 3.2 문자열 → Enum 전환의 파급 효과
- **해결**: `build.rs`에서 TOML 데이터로부터 enum을 자동 생성. 병렬 인덱스 전략 적용.

### 3.3 시스템 모듈 트리 재구성
- **해결**: 9개 도메인별 서브디렉토리(combat/ai/item/creature/world/social/spawn/identity/misc/)로 재편.

### 3.4 unwrap() 전면 제거
- **방법**: `if let Some`, `unwrap_or_else`, `match`, `thiserror` 기반 `GameError` enum 활용.

## 10. 안정화(Stabilization) 교훈 (v2.41.1)

### 10.1 ECS의 리소스 이중화(Dual State)는 컴파일러가 잡지 못한다
- **교훈**: `resources.insert(grid.clone())`으로 Grid를 Resources에 넣은 후, 시스템이 수정한 복사본을 원본으로 역동기화하지 않으면 **렌더러만 보는 원본**과 **시스템이 보는 복사본**이 영원히 분리된다.
- **핵심**: Rust의 소유권 시스템은 `clone()`으로 만든 두 인스턴스를 서로 다른 객체로 인식하여 **동기화 누락을 감지하지 못한다**.
- **해결법**: 시스템 실행 후 역동기화 코드를 반드시 삽입하거나, 처음부터 단일 소스(Single Source of Truth) 패턴 적용.

### 10.2 테스트 전량 통과 ≠ 플레이 가능
- **교훈**: 4,177개 테스트가 전부 통과해도 실제 게임 루프에서 전혀 작동하지 않을 수 있다. 단위 테스트는 개별 시스템의 순수 함수만 검증하지, **시스템 간 데이터 흐름**은 검증하지 않는다.
- **해결법**: E2E 플레이 테스트를 최우선으로 두고, 단위 테스트는 보조 수단으로 활용.

### 10.3 초기 장비 미장착 = 즉사
- **교훈**: AC 10(갑옷 없음) + 맨손(1-2 데미지)으로는 NetHack 1층에서도 생존 불가. 원본은 `ini_inv()`에서 자동으로 시작 장비를 장착함.
- **해결법**: 인벤토리에 넣는 것과 장착하는 것은 별개 작업. `Equipment.slots.insert()` + `Inventory.remove_item()` 쌍을 반드시 함께 호출.

### 10.4 리소스 이중화 감사 방법론
- **방법**: (1) `#[resource] xxx: &mut Type`을 사용하는 모든 시스템을 grep으로 수집 (2) 렌더러/UI에서 해당 타입을 `self.game.*`로 접근하는지 vs `resources.get()`으로 접근하는지 대조 (3) 패턴이 다르면 역동기화 필요.
